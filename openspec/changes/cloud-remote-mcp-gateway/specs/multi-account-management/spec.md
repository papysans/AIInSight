## ADDED Requirements

### Requirement: Each skill user SHALL have an isolated account with independent XHS credentials
The system SHALL associate each API key with a distinct account, and each account SHALL maintain its own XHS login session, cookies, and QR authentication flow independently of all other accounts. XHS account isolation SHALL leverage ShunL xhs-mcp's native multi-account capabilities (as established in `shunl-xhs-mcp-integration` change): the `account` parameter on all ShunL MCP calls (`xhs_add_account`, `xhs_publish_content`, `xhs_check_auth_status`, etc.) and SQLite per-account session persistence.

#### Scenario: Two users log into XHS independently
- **WHEN** user A completes XHS QR login and user B initiates their own XHS QR login
- **THEN** user B's login flow MUST NOT invalidate or modify user A's existing session, because each user maps to a separate `account` in the ShunL sidecar

#### Scenario: User resets their XHS login
- **WHEN** a user resets their own XHS login state
- **THEN** only that user's account in the ShunL sidecar MUST be affected; other accounts MUST remain unchanged

### Requirement: XHS account_id mapping SHALL flow through the full call chain
The `account_id` extracted from the API key at the Gateway layer SHALL be propagated through `XiaohongshuPublisher` to all ShunL MCP calls as the `account` parameter, ensuring that XHS operations are always scoped to the correct user.

#### Scenario: User publishes to XHS
- **WHEN** user A calls `publish_to_xhs` through the Gateway
- **THEN** the `XiaohongshuPublisher` MUST call `xhs_publish_content` with `account` set to user A's account identifier in the ShunL sidecar

### Requirement: User settings SHALL be isolated per account
LLM API key configurations, agent provider overrides, Volcengine credentials, and all other user-configurable settings SHALL be stored and retrieved per account, not shared globally.

#### Scenario: User A configures custom LLM keys
- **WHEN** user A saves custom DeepSeek API keys via settings
- **THEN** those keys MUST only be used for user A's analysis tasks, not for any other account's tasks

### Requirement: Output artifacts SHALL be namespaced by account
All generated output artifacts (analysis markdown, card preview images, XHS login QR codes, image cache) SHALL be stored in account-scoped paths and SHALL only be accessible to the owning account.

#### Scenario: User queries their analysis outputs
- **WHEN** a user lists or downloads their analysis outputs
- **THEN** the response MUST only include artifacts generated by that user's account

### Requirement: API key management SHALL support account provisioning
The system SHALL provide a mechanism to create API keys for new accounts and to revoke/rotate existing keys, so that the operator can manage who has access to the cloud-hosted service.

#### Scenario: Operator provisions a new user
- **WHEN** the operator creates a new API key
- **THEN** the system MUST create a corresponding account with an empty, isolated state (no jobs, no XHS session, no settings)
