# Claude Skills 功能总览

> 面向 Claude Code / Claude CLI 用户的统一说明文档。本文档总结了当前项目中可用的 skill、MCP 工具、卡片能力、典型用户故事和推荐使用方式。

---

## 1. 文档目标

本项目当前已经具备一套可在 Claude 中直接调用的内容生产链路，目标是让用户只通过自然语言完成以下动作：

- 查看今日 AI 热点榜单
- 选择某条热点做深度分析
- 对任意 AI 话题发起多来源证据深挖
- 将单条热点生成小红书图文
- 将今日整榜生成小红书图文
- 在预览后确认发布到小红书

> 说明：本文中凡提到旧的 `opinion-analyzer`、`platforms`、国内平台抓取或登录容器，均属于历史设计。当前单话题链路应以 `ai-topic-analyzer`、AI 来源组和 evidence-first 工作流为准。

系统设计上分为三层：

- `Skill`：定义 Claude 在不同场景下应该走什么流程
- `MCP`：向 Claude 暴露可调用工具
- `Backend / Renderer / Publisher`：真正完成采集、分析、卡片渲染和发布

## 2. 当前可用 Skills

当前项目中主要有两个面向用户的 skill。

### 2.1 `ai-insight`

文件位置：
[`SKILL.md`](/Volumes/Work/Projects/AIInSight/.agents/skills/ai-insight/SKILL.md)

定位：

- 面向 AI 日报和 AI 热点榜单
- 负责从多源采集 AI 热点
- 支持单话题分析
- 支持单条热点卡片生成与发布
- 支持今日整榜卡片生成与发布

典型触发语：

- `今日AI有什么热点？`
- `我想看一下当前的用户热点`
- `分析 1`
- `把第 1 条做成小红书图文`
- `把今天的榜单做成小红书图文`
- `把今天的榜单直接发布到小红书`

### 2.2 `ai-topic-analyzer`

文件位置：
[`SKILL.md`](/Volumes/Work/Projects/AIInSight/.agents/skills/ai-topic-analyzer/SKILL.md)

定位：

- 面向任意 AI 话题、模型、公司、论文或项目
- 走多来源证据检索 + 辩论式分析链路
- 默认直接分析，不再询问平台
- 先分析，再生成卡片，再发布
- 支持发布到小红书

典型触发语：

- `帮我看看“xxx”这个 AI 话题`
- `分析这个话题`
- `深挖这个模型/项目`
- `给这个分析生成卡片`
- `把这个分析发布到小红书`

## 3. 系统能力全景

### 3.1 AI 日报链路

适合“今天 AI 圈发生了什么”这类需求。

核心能力：

- 从多源采集 AI 热点
- 聚合、去重、打分、排序
- 返回今日 Top 榜单
- 对单个话题做进一步分析
- 生成单条热点图文
- 生成今日整榜图文
- 发布到小红书

默认使用的数据源包括：

- `aibase`
- `jiqizhixin`
- `qbitai`
- `github_trending`
- `producthunt_ai`
- `hf_papers`
- `techcrunch_ai`

### 3.2 热点辩论链路

适合“帮我看看某个热点的舆论走向”这类需求。

核心能力：

- 多平台抓取热点内容
- 启动后台分析任务
- 轮询任务状态
- 输出总结、洞察、标题、副标题和文案
- 生成结构化分析卡片
- 输出 AI 配图
- 发布到小红书

默认可分析平台：

- `wb`
- `dy`
- `ks`
- `bili`

可扩展平台包括：

- `tieba`
- `zhihu`
- `xhs`
- `hn`
- `reddit`

## 4. 用户故事

这一节从真实使用场景出发，描述系统已经支持的典型用户故事。

### 4.1 用户故事 A：查看今日 AI 热点榜单

作为内容创作者，我希望直接在 Claude 里说一句话，就能看到今天最值得关注的 AI 热点列表，方便我快速判断哪些话题值得做内容。

推荐输入：

```text
今日AI有什么热点？
```

预期系统行为：

1. Claude 命中 `ai-insight`
2. 调用 `get_ai_daily`
3. 返回榜单列表
4. 展示每条热点的标题、来源数、标签和采集时间
5. 提示用户可以继续分析某个编号

### 4.2 用户故事 B：分析单条 AI 热点

作为研究者，我希望在看完榜单后，直接对某一条热点做更深入的分析，而不是只看标题。

推荐输入：

```text
分析 5
```

预期系统行为：

1. Claude 识别用户选择了榜单中的某个话题
2. 询问分析深度：`quick` / `standard` / `deep`
3. 调用 `analyze_ai_topic`
4. 返回话题解读、趋势判断和摘要
5. 进一步询问是否需要生成图文或发布

### 4.3 用户故事 C：将单条 AI 热点做成小红书图文

作为运营人员，我希望把榜单中的某一条热点快速变成一套可发布的小红书图文，而不是手动拼图和写文案。

推荐输入：

```text
把第 1 条做成小红书图文，先不要发布
```

预期系统行为：

1. Claude 复用已经拿到的话题上下文
2. 调用 `generate_ai_daily_cards`
3. 默认生成单话题卡片包
4. 返回已生成的卡片类型
5. 等待用户继续确认是否发布

### 4.4 用户故事 D：直接发布单条 AI 热点到小红书

作为创作者，我希望在看到预览或确认内容后，一句话完成发布。

推荐输入：

```text
把第 1 条做成小红书图文并发布
```

或：

```text
确认发布
```

预期系统行为：

1. 如尚未生成卡片，先调用 `generate_ai_daily_cards`
2. 调用 `publish_ai_daily`
3. 返回发布结果
4. 如果成功，返回笔记链接或发布状态

### 4.5 用户故事 E：将今日整榜做成小红书图文

作为 AI 资讯号运营者，我希望把今天的 AI 榜单整体做成一篇榜单型内容，而不是只能发单条热点。

推荐输入：

```text
把今天的榜单做成小红书图文，先不要发布
```

预期系统行为：

1. Claude 识别这是“整榜”需求
2. 调用 `generate_ai_daily_ranking_cards`
3. 默认生成榜单型卡片包
4. 返回已生成卡片信息
5. 用户确认后可继续发布

### 4.6 用户故事 F：直接发布今日整榜到小红书

作为内容运营，我希望把今天的 AI 热点整榜一次性发布出去。

推荐输入：

```text
把今天的榜单直接发布到小红书
```

预期系统行为：

1. Claude 先要求用户确认
2. 调用 `publish_ai_daily_ranking`
3. 后端自动生成榜单标题、正文、标签和卡片
4. 发布到小红书
5. 返回结果

### 4.7 用户故事 G：分析任意热点并走辩论链路

作为舆情分析师，我希望输入任何一个热点话题，让系统自动去多个平台抓取数据、做结构化分析，并输出可发布内容。

推荐输入：

```text
帮我看看“DeepSeek”这个热点
```

预期系统行为：

1. Claude 命中 `ai-topic-analyzer`
2. 询问平台、图片数量、辩论轮次
3. 若用户回复 `默认`，使用默认配置
4. 调用 `analyze_topic`
5. 持续轮询 `get_analysis_status`
6. 完成后调用 `get_analysis_result`
7. 展示结果并等待发布确认

### 4.8 用户故事 H：发布热点辩论结果到小红书

作为热点内容创作者，我希望系统先给我完整预览，再由我决定是否发布。

推荐输入：

```text
发布
```

预期系统行为：

1. 只有在 `get_analysis_result` 已经拿到结果后才允许继续
2. Claude 二次确认
3. 调用 `publish_to_xhs`
4. 返回发布结果

### 5.1 AI 日报相关工具

## 6. Card 体系

当前渲染器一共支持 6 种 card。

渲染注册位置：
[`render-entry.js`](/Volumes/Work/Projects/GlobalInSight/renderer/render-entry.js)

### 6.1 `title`

用途：

- 作为封面卡
- 用于单条热点或整榜内容的开场图

适用链路：

- AI 日报单话题
- AI 日报整榜
- 热点辩论

### 6.2 `hot-topic`

用途：

- 用于展示单条 AI 热点的详细信息
- 包含标题、摘要、标签、来源数和热度分

适用链路：

- AI 日报单话题

### 6.3 `daily-rank`

用途：

- 用于榜单型内容
- 展示 Top N 排名、标题、分数和标签

适用链路：

- AI 日报整榜
- AI 日报单话题的“榜单式单卡”表达

### 6.4 `radar`

用途：

- 展示平台覆盖或多维对比

适用链路：

- 热点辩论

### 6.5 `timeline`

用途：

- 展示辩论过程或观点演化时间线

适用链路：

- 热点辩论

### 6.6 `trend`

用途：

- 展示趋势阶段、增长幅度和趋势曲线

适用链路：

- 热点辩论

## 8. Skill 行为约束

这部分是系统可用性最关键的地方。

### 8.1 `ai-insight` 的行为约束

- 默认优先使用缓存，不主动强刷
- 获取 AI 日报后，必须先展示榜单
- 用户选择分析时，需要确认分析深度
- 发布前必须明确确认
- 如果已经生成过卡片，发布时优先复用，不重复生成

### 8.2 `ai-topic-analyzer` 的行为约束

- 启动分析前，先确认平台、图片数量、辩论轮次
- `analyze_topic` 成功后，必须持续轮询 `get_analysis_status`
- 任务完成后，必须调用 `get_analysis_result`
- 展示结果后再问用户是否发布
- 未经确认绝不调用 `publish_to_xhs`
- 如果任务已经发布或发布失败，不应重新发起分析任务

## 12. 一句话总结

当前项目已经具备两类核心 skill：

- `ai-insight`：把“今日 AI 榜单”和“单条 AI 热点”变成可分析、可出图、可发布的小红书内容
- `ai-topic-analyzer`：把“任意 AI 话题”变成可分析、可解释、可发布的小红书内容

当前渲染器共支持 6 种 card：

- `title`
- `hot-topic`
- `daily-rank`
- `radar`
- `timeline`
- `trend`

从用户体验上看，系统已经可以支持两种成熟工作流：

1. `看榜单 -> 选热点 -> 生成图文 -> 发布`
2. `输入热点 -> 多平台分析 -> 生成图文 -> 发布`
