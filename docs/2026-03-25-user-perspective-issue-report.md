# 2026-03-25 用户视角问题文档

## 背景

本次链路覆盖了两个阶段：

1. 今日 AI 热点榜单图片渲染
2. 小红书榜单发布

这份文档只站在用户视角记录问题，不替系统找借口。重点是：

- 用户实际感受到哪里不顺
- 每个问题背后的真实原因
- 后续在另一侧应该如何修

---

## 一、用户视角下最明显的问题

### 1. 生图前的来回确认过多，体验偏“墨迹”

#### 用户感受

用户已经明确说了“帮我渲染小红书的图片”，系统却没有直接开干，而是进入了多轮确认：

- 先弹出 visual companion 提示
- 再问要哪种图片
- 再给 A/B/C 方案
- 再重复确认设计
- 再说开始渲染

从用户角度，这些步骤明显过多，尤其是在仓库已经有现成 `title + daily-rank` 渲染能力的前提下，会让人感觉“不直接”“拖沓”。

#### 准确判断

这不是用户需求不清晰，而是系统流程过度设计。

#### 根因

- 系统把一个本来可以“直接执行”的任务，当成了完整设计流程来走
- 触发了不必要的 brainstorming / plan / design 文档动作
- 没有针对“已有默认方案且风险极低”的场景做快速路径

#### 建议修法

- 对“帮我渲染图片”“帮我出图”这类明确指令，默认直接使用仓库现成方案执行
- 只有在以下情况下才追问：
  - 样式差异会明显影响结果
  - 存在多个完全不同的发布目标
  - 缺关键输入
- 默认策略建议：
  - 小红书整榜图片 → 直接走 `title + daily-rank`
  - 先渲染，渲染完再给用户看是否要重出，而不是渲染前反复问

#### 优先级

高。因为这是第一印象问题，直接影响“好不好用”。

---

### 2. 在简单任务里插入“设计文档 / 计划文档”痕迹，用户会感到不必要

#### 用户感受

用户只是要图片，但系统中途实际走了设计文档、执行计划、技能链路，这些对最终用户价值很低，还会拉长响应时间。

#### 根因

- 系统内部流程没有区分“一次性执行任务”和“需要长期维护的功能开发”
- 把工程化规范套到了即时操作任务上

#### 建议修法

- 新增轻量模式：
  - 如果任务是一次性执行、无代码改动、已有现成功能，则跳过 spec/plan
- 仅在下面场景保留完整设计流程：
  - 需要改代码
  - 需要改交互
  - 需要引入新模板或新发布流程

#### 优先级

中高。不是功能错误，但明显拖慢体验。

## 二、发布阶段暴露出的技术问题

### 3. 登录状态与发布账号不一致，导致首次发布失败

#### 用户看到的现象

系统一开始判断“小红书已登录”，但真正发布时却失败，报错：

```text
Account not found: _default
```

对用户来说，这非常困惑，因为表面上看是“明明登录了，为什么还不能发”。

#### 准确根因

这是典型的“登录态存在，但不是当前发布账号”的问题。

具体表现是：

- `check_xhs_status()` 在没有显式 `account_id` 时，检查的是默认上下文或某个已有浏览器态
- `publish_xhs_note()` 实际发布时使用了 `_default` 或错误账号上下文
- `xhs-mcp` 的多账号机制要求“登录、状态检查、发布”三步必须使用同一个账号 ID

也就是说：

- “系统里有某个账号登录了”
- 不等于
- “当前要发布的账号已经登录了”

#### 解决方法

需要强制把账号贯穿整条链路：

1. `get_xhs_login_qrcode(account_id=...)`
2. `check_xhs_login_session(session_id, account_id=...)`
3. `check_xhs_status(account_id=...)`
4. `publish_xhs_note(..., account_id=...)`

并且在扫码成功后，要把返回的真实账号 ID 持久化，用这个账号 ID 做后续发布，而不是再退回 `_default`。

#### 建议落地修复

- 扫码成功后，把 `session -> account_id` 映射保存下来
- 发布前不要再猜 `_default`
- 如果当前会话对应账号和发布账号不一致，明确提示：
  - “当前登录的是 A，发布目标是 B，请重新扫码”
- 前端/技能层不要把 `_default` 暴露成默认发布目标

#### 优先级

最高。这是首个真实阻断发布的问题。

### 4. 二维码展示能力不够好，用户很容易错过

#### 用户看到的现象

用户反馈“刚刚没看到二维码”，需要重新生成一次。

这说明当前二维码展示方式不够稳。

#### 准确根因

本次链路里有两个明显问题：

1. ASCII 二维码没有显示出来
   - 日志里有：`No module named 'PIL'`
   - 说明二维码 ASCII 渲染依赖缺失

2. 只能依赖外部图片链接查看二维码
   - 用户如果没有第一时间看到，体验会断掉
   - 二维码又有过期时间，容易错失窗口

#### 解决方法

- 安装二维码 ASCII 渲染所需依赖，确保终端里能直接看到码
- 即使 ASCII 失败，也要清晰输出：
  - 二维码图片链接
  - 会话 ID
  - 过期时间
  - “扫完回复已扫码”
- 用户说“重新生成二维码”时，直接生成新码，不要走额外确认

#### 建议落地修复

- 修复 `_generate_ascii_qr` 的依赖问题
- 在 CLI 中把二维码信息做成固定结构展示
- 把有效期从用户角度写清楚，比如“4 分钟内有效”

#### 优先级

高。它不一定是底层大 bug，但会显著影响登录成功率。

---

### 5. 同一个账号下，登录检查结果前后不一致或不稳定

#### 用户看到的现象

在发布过程中出现过几种不一致表现：

- 一次检查说已登录
- 另一次检查超时
- 再换具体账号检查时，又显示未登录

对用户来说，这会让系统显得“不靠谱”。

#### 准确根因

这不是单一 bug，更像是三件事叠加：

1. 默认账号 `_default` 和真实发布账号混用
2. `check_xhs_status()` 可能命中旧浏览器态
3. 浏览器驱动的状态检查本身有超时波动

#### 解决方法

- 所有状态检查都显式带 `account_id`
- 发布前只认“目标账号”的状态，不认全局状态
- 对状态检查增加“超时但不丢上下文”的重试策略

#### 建议落地修复

- UI/技能层展示：
  - 当前发布账号 ID
  - 当前发布账号昵称
  - 当前账号登录状态
- 不再返回模糊的“系统已登录”结论

#### 优先级

高。会影响用户对整条发布链路的信任。

### 6. 发布时图片路径不可见，导致第二次发布失败

#### 用户看到的现象

扫码成功后再次发布，系统报错：

```text
No valid image files found
```

这对用户来说是很典型的“明明图已经生成了，为什么发的时候又说没图”。

#### 准确根因

这次失败不是内容问题，而是容器之间的文件路径不通。

本次实际传给发布器的是：

- `mcp` 容器里的渲染产物路径：`/app/outputs/card_previews/...`

但真正执行发布的是 `xhs-mcp` 容器，它能访问的是共享卷路径：

- `mcp` 侧写入：`/app/runtime/xhs/images/...`
- `xhs-mcp` 侧读取：`/app/images/...`

所以：

- 图确实存在
- 只是发布容器看不到传入的那条路径

#### 解决方法

本次已经验证可行的解法是：

1. 先把图片转成 `data:image/png;base64,...`
2. 让发布层通过 `_process_image()` 自动写入共享卷
3. 再把共享卷里的 `/app/images/...` 路径传给 `xhs_publish_content`

#### 建议落地修复

最佳做法不是让上层自己转 data URL，而是让发布层自动处理这三类输入：

- `outputs/card_previews/...` 本地预览路径
- HTTP 预览 URL
- data URL

统一在 `publish_xhs_note()` / `_process_image()` 里落到共享卷，再交给 `xhs-mcp`。

换句话说，调用方不应该关心容器内部路径转换。

#### 优先级

最高。这是第二个真实阻断发布的问题。

### 7. 二维码为账号维度生成，但实际文件目录没有体现 account_id

#### 现象

按设计文档，二维码元数据和文件应该支持账号隔离；但本次实际返回的二维码路径是：

```text
/app/outputs/xhs_login/xhs-login-qrcode-xxxx.png
```

不是预期的账号子目录形式。

#### 准确判断

这说明“账号隔离设计”与“实际二维码文件落盘路径”可能还没完全对齐。

#### 可能根因

在 `get_login_qrcode()` 中，二维码输出目录的调用看起来没有把 `account_id` 传进去，导致图片文件仍然落在公共目录，而元数据却可能按账号维度保存。

#### 风险

- 多账号下二维码文件可能混放
- 排查登录问题时不够直观
- 后续清理过期二维码时容易误删或误用

#### 建议修法

- 二维码图片和 metadata 一律使用 `outputs/xhs_login/{account_id}/`
- 返回给上层的 `qr_image_path` / `latest.json` 路径也保持同一账号目录

#### 优先级

中高。当前不一定直接阻断，但会放大多账号排障难度。

### 8. 发布成功了，但没有拿到笔记链接

#### 用户看到的现象

最终发布成功，但返回结果是：

```json
{"success": true, "note_url": null}
```

这意味着用户虽然“发出去了”，但系统没法立即给出可点击的帖子链接。

#### 准确根因

当前上游 `xhs_publish_content` 返回结果不稳定，有时不带 `note_url` 或 `post_id`。

适配层已经支持从返回数据里尝试提取链接，但这次上游没有提供，所以最终只能返回 `null`。

#### 解决方法

有两条路径：

1. 优先修上游返回结构，让发布接口稳定返回 `note_url` / `post_id`
2. 如果上游拿不到，就在发布成功后追加一次查询：
   - 按最新发布内容
   - 或按账号最近笔记列表
   - 反查刚发布的帖子链接

#### 建议落地修复

- 在 `publish_xhs_note()` 成功后追加 post-publish resolve 步骤
- 如果确实拿不到链接，也要明确返回：
  - “已发布成功，但上游未返回 note_url，请在小红书 App 内查看”

#### 优先级

中高。不是阻断型 bug，但会明显影响“发布成功”的可验证性。

## 三、建议的修复优先级

### P0：必须先修

1. 发布账号与登录账号不一致，导致 `_default` / 错账号发布失败

### P1：强烈建议尽快修

3. 生图流程询问过多，默认不够直接
4. 二维码展示不友好，用户容易错过
5. 状态检查结果不稳定，默认账号和真实账号混用

### P2：建议完善

6. 二维码文件目录未完全账号隔离
7. 发布成功但拿不到 `note_url`
8. 一次性操作里不应强制走完整 spec/plan 流程

## 四、推荐改造方案（可直接转给另一边）

### 方案 1：给“出图”加快速路径

目标：用户说“帮我渲染小红书图片”时，直接出图，不要再走多轮设计确认。

建议：

- 默认卡片组合：`title + daily-rank`
- 默认主题：`warm`
- 先出图，后微调
- 只有缺必要输入时才追问

---

### 方案 2：把 XHS 登录/发布做成真正的账号闭环

目标：同一账号完成“生成码 → 扫码 → 状态检查 → 发布”。

建议：

- 生成二维码时创建 `account_id`
- 扫码成功后，持久化 `session_id -> account_id`
- 所有发布相关操作必须带同一个 `account_id`
- 不再默认回退 `_default`

---

### 方案 3：统一图片发布输入层

目标：调用方不用再关心容器路径。

建议：

- `publish_xhs_note()` 接受任何一种图片输入：
  - 本地卡片路径
  - HTTP 图片 URL
  - data URL
- 内部统一转换到共享卷路径，再交给 `xhs-mcp`

---

### 方案 4：增强发布成功后的可验证性

目标：发布成功后最好能给用户一个明确的结果链接。

建议：

- 发布成功后追加一次帖子解析/回查
- 优先拿 `note_url`
- 拿不到时也要明确告诉用户“已发出，但链接暂不可得”

## 六、结论

这次不是单点故障，而是：

- 前半段有“流程太重”的体验问题
- 后半段有“多账号发布链路还没完全打磨好”的工程问题

好消息是，核心路径已经打通：

- 图能渲染
- 二维码能生成
- 扫码登录能成功
- 通过共享卷转换后，榜单已经成功发布

因此下一步不是重做，而是把上面这些断点一个个抹平，让用户感受到的是“直接、稳定、一次成功”。
